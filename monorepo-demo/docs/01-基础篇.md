## 什么是 Monorepo

Monorepo(单一仓库)是一种代码管理策略,将多个相关的项目放在一个 Git 仓库中管理。

### 对比 Polyrepo

| 特性 | Monorepo | Polyrepo |
|------|----------|----------|
| 仓库数量 | 1 个 | 多个 |
| 代码共享 | 容易 | 困难 |
| 依赖管理 | 统一 | 分散 |
| 工具配置 | 统一 | 各自配置 |
| 原子提交 | 支持 | 不支持 |

### 典型结构

\`\`\`
monorepo-demo/
├── packages/
│   ├── shared/      # 共享包
│   ├── config/      # 配置包
│   └── apps/        # 应用包
├── package.json     # 根配置
└── pnpm-workspace.yaml
\`\`\`

## 为什么使用 Monorepo

### 优势

1. **代码复用**
   - 共享组件、工具函数、类型定义
   - 避免重复代码

2. **统一管理**
   - 统一的依赖版本
   - 统一的代码规范
   - 统一的构建流程

3. **原子提交**
   - 一次提交可以修改多个包
   - 保证代码一致性

4. **重构方便**
   - 跨包重构更简单
   - IDE 支持更好

5. **简化工作流**
   - 一次性安装所有依赖
   - 统一的测试和发布

### 劣势

1. **仓库体积大**: 随着项目增多,仓库会变大
2. **构建时间**: 需要优化构建流程
3. **权限管理**: 所有人都能看到所有代码
4. **工具链要求**: 需要 Monorepo 工具支持

## pnpm Workspace 基础

### pnpm-workspace.yaml

这是 pnpm workspace 的配置文件:

\`\`\`yaml
packages:
  - 'packages/shared/*'
  - 'packages/config/*'
  - 'packages/apps/*'
\`\`\`

- 使用 glob 模式匹配包目录
- 每个匹配的目录必须有 `package.json`

### .npmrc 配置

\`\`\`ini
# 防止幽灵依赖
hoist=false

# 提升某些包
public-hoist-pattern[]=*typescript*

# 自动安装 peer dependencies
auto-install-peers=true
\`\`\`

### 什么是幽灵依赖?

幽灵依赖是指:代码中使用了未在 `package.json` 中声明的依赖。

**示例**:

\`\`\`json
// package.json 只声明了 react
{
  "dependencies": {
    "react": "^18.0.0"
  }
}
\`\`\`

\`\`\`typescript
// 但代码中使用了 lodash(react 的依赖)
import _ from 'lodash'; // 幽灵依赖!
\`\`\`

**为什么会出现?**

npm/yarn 会将依赖提升到根 `node_modules`,导致可以访问未声明的包。

**如何避免?**

pnpm 的 `hoist=false` 可以防止提升,强制显式声明依赖。

## 包管理与依赖

### 包命名规范

使用 `@scope/package-name` 格式:

\`\`\`json
{
  "name": "@demo/utils"
}
\`\`\`

- `@demo`: 组织名(scope)
- `utils`: 包名

### Workspace 协议

引用本地包使用 `workspace:*`:

\`\`\`json
{
  "dependencies": {
    "@demo/utils": "workspace:*",
    "@demo/types": "workspace:^"
  }
}
\`\`\`

- `workspace:*`: 链接到 workspace 中的最新版本
- `workspace:^`: 链接并在发布时替换为语义化版本

### 添加依赖

\`\`\`bash
# 为特定包添加依赖
pnpm --filter @demo/web-app add react

# 为所有包添加开发依赖
pnpm -r add -D typescript

# 在根目录添加(所有包共享)
pnpm add -Dw eslint
\`\`\`

### 依赖类型

1. **dependencies**: 运行时依赖
2. **devDependencies**: 开发时依赖
3. **peerDependencies**: 对等依赖(由使用方安装)

**示例**:

\`\`\`json
{
  "name": "@demo/ui-react",
  "peerDependencies": {
    "react": ">=18.0.0"  // 使用方需要安装 react
  },
  "dependencies": {
    "classnames": "^2.5.1"  // 组件库自己依赖
  }
}
\`\`\`

## 脚本命令

### 根目录脚本

\`\`\`json
{
  "scripts": {
    "dev": "pnpm --filter \"./packages/apps/*\" dev",
    "build": "pnpm -r build",
    "lint": "pnpm -r lint"
  }
}
\`\`\`

### 常用命令参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `-r` / `--recursive` | 在所有包中执行 | `pnpm -r build` |
| `--filter <pattern>` | 筛选特定包 | `pnpm --filter @demo/web-app dev` |
| `-w` / `--workspace-root` | 在根目录执行 | `pnpm add -Dw typescript` |
| `--parallel` | 并行执行 | `pnpm --parallel -r build` |

### Filter 模式

\`\`\`bash
# 单个包
pnpm --filter @demo/utils build

# Glob 模式
pnpm --filter \"@demo/*\" build

# 目录模式
pnpm --filter \"./packages/apps/*\" dev

# 依赖图模式
pnpm --filter @demo/web-app... build  # web-app 及其依赖
pnpm --filter ...@demo/utils build    # utils 及其依赖者
\`\`\`

## 常见问题

### Q1: 为什么选择 pnpm 而不是 npm/yarn?

**A**: pnpm 的优势:
- ⚡ 更快的安装速度
- 💾 节省磁盘空间(通过硬链接)
- 🔒 更严格的依赖管理(防止幽灵依赖)
- 📦 原生支持 workspace

### Q2: workspace 协议在发布时会怎样?

**A**: 
\`\`\`json
// 发布前
{
  "dependencies": {
    "@demo/utils": "workspace:^"
  }
}

// 发布后(自动替换)
{
  "dependencies": {
    "@demo/utils": "^1.0.0"
  }
}
\`\`\`

### Q3: 如何调试 workspace 中的包?

**A**: 
1. pnpm 会创建 symlink,修改源码立即生效
2. 使用 `pnpm link --global` 全局链接
3. 在 VSCode 中配置多根工作区

### Q4: 构建顺序如何控制?

**A**: 
pnpm 会根据依赖关系自动确定构建顺序。也可以使用 Turborepo 优化。

### 练习 1: 创建新包

创建一个新的工具包 `@demo/helpers`:

1. 在 `packages/shared` 下创建 `helpers` 目录
2. 初始化 `package.json`
3. 添加一个简单的工具函数
4. 在 `web-app` 中引用并使用

### 练习 2: 依赖管理

为 `@demo/utils` 添加 `lodash-es` 依赖:

1. 使用正确的 pnpm 命令添加
2. 在代码中使用 lodash 函数
3. 验证是否工作

### 练习 3: 脚本编写

在根 `package.json` 中添加脚本:

1. `dev:all`: 同时启动所有应用
2. `build:libs`: 只构建共享库
3. `clean:all`: 清理所有构建产物

### 练习 4: 问题排查

遇到以下问题该如何解决?

1. 安装依赖后找不到模块
2. TypeScript 报错找不到类型
3. 循环依赖问题

### 练习 1 答案

\`\`\`bash
# 1. 创建目录
mkdir -p packages/shared/helpers/src

# 2. 创建 package.json
cat > packages/shared/helpers/package.json <<EOF
{
  "name": "@demo/helpers",
  "version": "1.0.0",
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts"
}
EOF

# 3. 创建工具函数
# src/index.ts
export const add = (a: number, b: number) => a + b;

# 4. 在 web-app 中使用
pnpm --filter @demo/web-app add @demo/helpers

# 在代码中
import { add } from '@demo/helpers';
\`\`\`

### 练习 2 答案

\`\`\`bash
# 添加依赖
pnpm --filter @demo/utils add lodash-es
pnpm --filter @demo/utils add -D @types/lodash-es

# 使用
import { debounce } from 'lodash-es';
\`\`\`

### 练习 3 答案

\`\`\`json
{
  "scripts": {
    "dev:all": "pnpm --parallel --filter \"./packages/apps/*\" dev",
    "build:libs": "pnpm --filter \"./packages/shared/*\" build",
    "clean:all": "pnpm -r clean"
  }
}
\`\`\`

### 练习 4 答案

1. **找不到模块**: 运行 `pnpm install` 重新安装
2. **类型错误**: 检查 `tsconfig.json` 中的 `paths` 配置
3. **循环依赖**: 重构代码,提取共同依赖到新包
